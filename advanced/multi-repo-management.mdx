The `/multi-repo` command provides sophisticated multi-repository coordination and management capabilities for enterprise-scale development environments with multiple teams, services, and dependencies.

## Understanding Multi-Repo Strategies

Modern software development often involves managing multiple repositories. Understanding the trade-offs between different strategies is crucial.

### Repository Strategies

<Tabs>
  <Tab title="Monorepo">
    **Single repository containing multiple projects**
    
    **Advantages:**
    - Atomic cross-project changes
    - Simplified dependency management
    - Unified tooling and CI/CD
    - Single source of truth
    - Easy code sharing
    
    **Challenges:**
    - Large repository size
    - Complex build systems
    - Access control complexity
    - Steep learning curve
    
    **Best for:**
    - Tightly coupled services
    - Shared code bases
    - Fast-moving teams
    - Internal tools
    
    **Example:**
    ```
    company-monorepo/
    ├── packages/
    │   ├── api/
    │   ├── web-app/
    │   ├── mobile-app/
    │   └── shared-lib/
    ├── tools/
    └── docs/
    ```
  </Tab>
  
  <Tab title="Polyrepo">
    **Multiple independent repositories**
    
    **Advantages:**
    - Independent versioning
    - Clear ownership boundaries
    - Smaller, focused repos
    - Simpler access control
    - Technology diversity
    
    **Challenges:**
    - Dependency coordination
    - Cross-repo changes complex
    - Tooling duplication
    - Version management overhead
    
    **Best for:**
    - Microservices
    - Independent teams
    - Third-party libraries
    - Open source projects
    
    **Example:**
    ```
    company-api/          (separate repo)
    company-web-app/      (separate repo)
    company-mobile-app/   (separate repo)
    company-shared-lib/   (separate repo)
    ```
  </Tab>
  
  <Tab title="Hybrid">
    **Mix of monorepo and polyrepo approaches**
    
    **Advantages:**
    - Flexibility
    - Best of both worlds
    - Gradual migration path
    - Team autonomy
    
    **Challenges:**
    - Complexity
    - Inconsistent tooling
    - Coordination overhead
    
    **Best for:**
    - Organizations in transition
    - Mixed ownership models
    - Varying coupling levels
    
    **Example:**
    ```
    core-monorepo/        (critical shared code)
    service-a/            (independent service)
    service-b/            (independent service)
    external-integrations/ (separate concerns)
    ```
  </Tab>
  
  <Tab title="Federated">
    **Distributed repositories with coordination layer**
    
    **Advantages:**
    - Team independence
    - Geographic distribution
    - Technology freedom
    - Scalability
    
    **Challenges:**
    - Coordination complexity
    - Tooling overhead
    - Discovery challenges
    
    **Best for:**
    - Global organizations
    - Acquisitions/mergers
    - Platform ecosystems
    
    **Example:**
    ```
    region-us/repos/      (US team repos)
    region-eu/repos/      (EU team repos)
    region-apac/repos/    (APAC team repos)
    platform-core/        (shared platform)
    ```
  </Tab>
</Tabs>

## Multi-Repo Command Usage

The `/multi-repo` command provides comprehensive repository coordination:

```bash
/multi-repo [action] [scope] [strategy] [parameters]
```

### Actions

<CardGroup cols={2}>
  <Card title="coordinate" icon="diagram-project">
    Coordinate changes across repositories
    ```bash
    /multi-repo coordinate organization polyrepo
    ```
  </Card>
  
  <Card title="sync" icon="arrows-rotate">
    Synchronize repository states
    ```bash
    /multi-repo sync team hybrid --branch-strategy=gitflow
    ```
  </Card>
  
  <Card title="migrate" icon="truck">

## Cross-Repository Coordination

### Coordinating Related Changes

When changes span multiple repositories:

<Steps>
  <Step title="Identify Dependencies">
    ```bash
    /multi-repo coordinate organization --analyze-impact
    ```
    
    **Output:**
    ```
    Impact Analysis:
    
    Primary Repository: api-gateway
    Change Type: Breaking API change
    
    Affected Repositories:
    ├── web-app (direct consumer)
    ├── mobile-app (direct consumer)
    ├── analytics-service (indirect via web-app)
    └── notification-service (direct consumer)
    
    Coordination Strategy:
    1. Update API contract in api-gateway
    2. Update clients: web-app, mobile-app, notification-service
    3. Deploy in order: gateway → services → clients
    4. Validate integration tests
    ```
  </Step>
  
  <Step title="Create Coordination Branch">
    ```bash
    /multi-repo coordinate project api-upgrade --create-branches
    ```
    
    Creates coordinated branches across affected repos:
    ```
    Created branches:
    ✓ api-gateway: feature/api-v2
    ✓ web-app: feature/api-v2-client
    ✓ mobile-app: feature/api-v2-client
    ✓ notification-service: feature/api-v2-client
    ```
  </Step>
  
  <Step title="Synchronize Changes">
    ```bash
    /multi-repo sync project api-upgrade --atomic
    ```
    
    Ensures all changes are synchronized:
    ```
    Synchronization Status:
    ✓ All repositories on feature/api-v2 branches
    ✓ Changes committed and pushed
    ✓ Integration tests passing
    ✓ Ready for coordinated merge
    ```
  </Step>
  
  <Step title="Coordinated Deployment">
    ```bash
    /multi-repo deploy project api-upgrade --order=dependency
    ```
    
    Deploys in dependency order:
    ```
    Deployment Order:
    1. api-gateway (no dependencies)
    2. notification-service (depends on gateway)
    3. web-app (depends on gateway)
    4. mobile-app (depends on gateway)
    
    ✓ All deployments successful
    ✓ Integration validated
    ```
  </Step>
</Steps>

### Dependency Management

<Accordion title="Shared Library Updates">
  **Scenario:** Update a shared library across all consuming repositories
  
  ```bash
  /multi-repo coordinate organization --update-dependency @company/shared-lib@2.0.0
  ```
  
  **Process:**
  1. Identify all repositories using `@company/shared-lib`
  2. Create update branches in each repository
  3. Update dependency and run tests
  4. Create PRs with compatibility reports
  5. Coordinate review and merge
  
  **Output:**
  ```
  Dependency Update: @company/shared-lib 1.5.0 → 2.0.0
  
  Affected Repositories: 23
  ├── Compatible (no changes needed): 15
  ├── Breaking changes (code updates needed): 6
  └── Testing required: 2
  
  Created Pull Requests:
  ├── api-gateway: #234 (breaking changes)
  ├── web-app: #456 (breaking changes)
  ├── data-processor: #789 (testing needed)
  └── ... (6 total)
  ```
</Accordion>

<Accordion title="Version Pinning Strategy">
  **Maintain consistent versions across repositories:**
  
  ```yaml
  # .multi-repo/dependency-policy.yml
  version_strategy: semver
  
  shared_dependencies:
    "@company/shared-lib":
      strategy: exact_version  # All repos use exact version
      version: "2.0.0"
      
    "@company/ui-components":
      strategy: minor_range    # Allow minor updates
      version: "^3.5.0"
      
    "react":
      strategy: major_range    # Allow within major version
      version: "~18.2.0"
  ```
  
  Enforce policy:
  ```bash
  /multi-repo govern organization --enforce-dependency-policy
  ```
</Accordion>

## Enterprise Governance

### Implementing Repository Governance

<Tabs>
  <Tab title="Access Control">
    **Role-Based Access Control (RBAC):**
    
    ```bash
    /multi-repo govern organization --implement-rbac
    ```
    
    ```yaml
    # .multi-repo/rbac-policy.yml
    roles:
      admin:
        permissions: [read, write, admin, delete]
        applies_to: all_repositories
        
      developer:
        permissions: [read, write]
        applies_to:
          - owned_repositories
          - team_repositories
        restrictions:
          - no_main_branch_push
          - require_pr_approval
          
      contractor:
        permissions: [read, write]
        applies_to: [assigned_repositories]
        restrictions:
          - no_secrets_access
          - no_production_deploy
          - time_limited_access
    
    teams:
      backend:
        repos: [api-*, *-service]
        role: developer
        
      frontend:
        repos: [web-*, mobile-*]
        role: developer
    ```

## Automation and Tooling

### Cross-Repository Automation

<Accordion title="Dependency Update Automation">
  **Automatic dependency updates across repositories:**
  
  ```bash
  /multi-repo automate organization --dependency-updates
  ```
  
  **Automation workflow:**
  1. Daily scan for dependency updates
  2. Create update branches in affected repos
  3. Run test suites automatically
  4. Create PRs with compatibility reports
  5. Auto-merge if tests pass (for minor/patch)
  6. Notify team for major updates
  
  **Configuration:**
  ```yaml
  automation:
    dependency_updates:
      schedule: daily
      auto_merge:
        patch: true
        minor: true
        major: false
      require_tests: true
      notify_on: [major_update, test_failure]
  ```
</Accordion>

<Accordion title="CI/CD Synchronization">
  **Sync CI/CD configurations:**
  
  ```bash
  /multi-repo sync organization --ci-config
  ```
  
  **Synchronizes:**
  - GitHub Actions workflows
  - Build configurations
  - Deployment pipelines
  - Test configurations
  - Environment variables (non-secret)
  
  **Template:**
  ```yaml
  # .multi-repo/ci-template.yml
  name: CI Pipeline
  
  on: [push, pull_request]
  
  jobs:
    test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v3
        - name: Setup
          run: {{ setup_command }}
        - name: Test
          run: {{ test_command }}
        - name: Coverage
          run: {{ coverage_command }}
  ```
</Accordion>

## Best Practices

### Repository Organization

<CardGroup cols={2}>
  <Card title="Clear Ownership" icon="user-check">

    - Each repository has a clear owner
    - CODEOWNERS file maintained

  <Card title="Consistent Structure" icon="folder-tree">

    - README with setup instructions

### Coordination Strategies

<Steps>
  <Step title="Plan Ahead">
    Coordinate breaking changes well in advance
    - Document breaking changes early
    - Communicate with affected teams
    - Plan migration timeline
    - Provide migration guides
  </Step>
  
  <Step title="Version Carefully">
    Use semantic versioning consistently
    - Major: Breaking changes
    - Minor: New features (backward compatible)
    - Patch: Bug fixes
    - Communicate version impacts
  </Step>
  
  <Step title="Test Thoroughly">
    Validate cross-repository changes
    - Integration test suites
    - Contract testing
    - End-to-end validation
    - Staging environment testing
  </Step>
  
  <Step title="Monitor Continuously">
    Track repository health and dependencies
    - Automated health checks
    - Dependency graphs
    - Security scanning
    - Performance metrics
  </Step>
</Steps>
