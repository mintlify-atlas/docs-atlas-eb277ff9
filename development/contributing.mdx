Thank you for your interest in contributing to ccprompts! This guide will help you get started with contributing to the project.

## How to Contribute

<Steps>
  <Step title="Fork and Clone">
    Fork the repository and clone it to your local machine:

    ```bash
    git clone https://github.com/YOUR_USERNAME/ccprompts.git
    cd ccprompts
    ```
  </Step>

  <Step title="Install Dependencies">
    Install Bun (package manager) and project dependencies:

    ```bash
    # Install Bun (macOS/Linux)
    curl -fsSL https://bun.sh/install | bash

    # Install dependencies
    bun install --frozen-lockfile
    ```

    <Note>
      This project uses Bun 1.1.34. Verify with `bun --version`.
    </Note>
  </Step>

  <Step title="Create a Branch">
    Create a new branch for your feature or fix:

    ```bash
    git checkout -b feature/my-new-feature
    # or
    git checkout -b fix/bug-description
    ```
  </Step>

  <Step title="Make Changes">
    Make your changes following the code style guidelines:

    - **Indentation**: 4 spaces (enforced by EditorConfig)
    - **Line length**: 120 characters for markdown
    - **Naming**: camelCase for variables, PascalCase for classes, kebab-case for files
    - **Comments**: Use JSDoc for all public methods
  </Step>

  <Step title="Test Your Changes">
    Run validation and tests before committing:

    ```bash
    # Run all tests
    bun run test

    # Run validation only
    bun run validate

    # Run linting
    bun run lint:fix
    ```
  </Step>

  <Step title="Commit Your Changes">
    Write clear, descriptive commit messages using conventional commits:

    ```bash
    git add .
    git commit -m "feat: add new command for X"
    # or
    git commit -m "fix: resolve issue with Y"
    # or
    git commit -m "docs: update README with Z"
    ```
  </Step>

  <Step title="Open a Pull Request">
    Push your branch and open a pull request:

    ```bash
    git push origin feature/my-new-feature
    ```

    Then open a pull request on GitHub with a detailed description of your changes.
  </Step>
</Steps>

## Code Style Guidelines

### TypeScript & JavaScript

- **Strict mode**: Always enabled in tsconfig.json
- **Target**: ES2020 with CommonJS modules
- **Imports**: External libraries first, then local modules
- **Types**: Use explicit typing, avoid `any`
- **Error handling**: Use try-catch with descriptive error messages

### Formatting

<CodeGroup>
```javascript Good Example
// 4-space indentation, camelCase variables
function validateCommand(commandPath) {
    const content = fs.readFileSync(commandPath, 'utf8');
    return content.length > 0;
}
```

```javascript Bad Example
// 2-space indentation, unclear naming
function validate(p) {
  const c = fs.readFileSync(p, 'utf8');
  return c.length > 0;
}
```
</CodeGroup>

### Naming Conventions

| Type | Convention | Example |
|------|-----------|----------|
| Files (scripts) | kebab-case | `validate-commands.js` |
| Files (TypeScript) | camelCase | `commandValidator.ts` |
| Variables | camelCase | `commandCount` |
| Classes | PascalCase | `CommandValidator` |
| Methods | camelCase | `validateFile()` |
| Constants | UPPER_SNAKE_CASE | `EXPECTED_COMMAND_COUNT` |

## Adding New Commands

When adding a new slash command to the collection:

<Steps>
  <Step title="Choose the Right Phase">
    Place your command in the appropriate phase directory:

    - `00-initial-workflow/` - Project analysis & intelligent chaining
    - `01-project-setup/` - Documentation, learning, MCP setup
    - `02-development/` - Backup, debug, migrate, optimize, refactor
    - `03-security/` - Security auditing & compliance
    - `04-testing/` - Testing & troubleshooting
    - `05-deployment/` - CI/CD & deployment
    - `06-collaboration/` - Code review & team workflows
    - `07-utilities/` - Development utilities & best practices
    - `08-extras/` - Health checks & modernization
    - `09-agentic-capabilities/` - MCP & agent orchestration
    - `10-ai-native-development/` - AI-powered development tools
    - `11-enterprise-scale/` - Governance & multi-repo
  </Step>

  <Step title="Follow the Command Structure">
    Create a markdown file with this structure:

    ```markdown
    # Command Name

    Short description of functionality

    ## Usage

    `/command-name`

    ## Description

    Detailed explanation with capabilities

    ## Example Scenarios

    Real-world use cases

    ## Safety Considerations

    Warnings, prerequisites, validation steps

    ## Related Commands

    Cross-references to complementary commands
    ```
  </Step>

  <Step title="Update Command Count">
    The validation system expects exactly 70 commands. If adding/removing commands:

    1. Update the count in `.claude-plugin/plugin.json`
    2. Update the `EXPECTED_COMMAND_COUNT` in your test runs
  </Step>

  <Step title="Validate">
    Run validation to ensure your command passes all checks:

    ```bash
    bun run validate
    bun run lint:fix
    ```
  </Step>
</Steps>

<Warning>
  If you add or remove command files, you **must** update the phase counts in `.claude-plugin/plugin.json` and ensure the total matches the expected count.
</Warning>

## Quality Gates

Before submitting a pull request, ensure all quality gates pass:

- [ ] `bun run validate` passes (70 commands found, no structural errors)
- [ ] `bun run lint` passes (no markdown style violations)
- [ ] `bun run check-links` passes (all links valid)
- [ ] `bun run test` passes if test files modified
- [ ] `bun run security-scan` checked for new security warnings
- [ ] Conventional commit message used (`feat:`, `fix:`, `docs:`, etc.)
- [ ] If adding/removing commands, `.claude-plugin/plugin.json` updated

## Reporting Issues

When reporting issues:

1. **Check existing issues** - Search for similar reports first
2. **Provide context** - Include your environment (Node version, OS, Bun version)
3. **Steps to reproduce** - Clear, numbered steps to reproduce the issue
4. **Expected vs actual** - What you expected vs what happened
5. **Error messages** - Include full error messages and stack traces

<Note>
  Open issues at [github.com/ursisterbtw/ccprompts/issues](https://github.com/ursisterbtw/ccprompts/issues)
</Note>

## Code of Conduct

Please be respectful and follow community guidelines:

- Be welcoming and inclusive
- Respect differing viewpoints and experiences
- Accept constructive criticism gracefully
- Focus on what's best for the community
- Show empathy towards other community members
