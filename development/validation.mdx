## Multi-Dimensional Validation

The validation engine validates across four dimensions:

### 1. Structural Validation

**What it checks:**
- Markdown format correctness
- Frontmatter presence and validity
- Section completeness (Usage, Description, Examples)
- Heading hierarchy
- Code block formatting
- Line endings (must be LF, not CRLF)

**Target:** 95%+ structural compliance

<Note>
  Each command receives a numeric `_score` field (0-100) reflecting validation pass rate.
</Note>

### 2. Security Validation

**Pattern detection:**

<CodeGroup>
```javascript Hardcoded Secrets
// Detects patterns like:
password = "actual-password-123"
api_key = "sk-1234567890abcdef"
secret = "my-secret-value"
token = "ghp_1234567890abcdef"
```

```javascript Dangerous Patterns
// Detects risky code:
eval(userInput)
element.innerHTML = userInput
`${user.input}` // template injection
```
</CodeGroup>

**Skip patterns:** Excludes examples with keywords like "example", "placeholder", "your-", "REPLACE_WITH"

### 3. Quality Scoring

**Quality checks:**

| Check | Weight | Description |
|-------|--------|-------------|
| Minimum content length | 15 | Content >500 characters |
| Examples section | 20 | Includes usage examples |
| Clear instructions | 25 | Has `<instructions>` tags |
| Security considerations | 15 | Mentions security/safety |
| Output requirements | 10 | Defines deliverables |
| Professional language | 10 | No TODO/FIXME markers |

**Scoring:** Each command gets a quality score of 0-100 based on checks passed.

### 4. Performance Metrics

**Targets:**
- File discovery: &lt;100ms
- Total validation: &lt;2000ms
- Current performance: ~3ms discovery + ~28ms validation

**Memory tracking:**
- Heap usage monitored
- Delta reported
- Average file processing time tracked

## Safety Validation System

The Dagger-based safety system provides containerized validation for dangerous commands.

### Running Safety Validation

```bash
# Run Dagger safety validation
bun run safety-validate
```

**What it validates:**

<Steps>
  <Step title="Pattern Detection">
    Identifies dangerous commands automatically via regex patterns:

    - `rm -rf` patterns
    - `curl | bash` patterns
    - `sudo` usage
    - `chmod` operations
    - `docker --privileged`
    - `eval()` calls
  </Step>

  <Step title="Container Isolation">
    Commands marked as dangerous execute in isolated Dagger containers:

    - Ubuntu 22.04 base image
    - Resource limits enforced (CPU/memory caps)
    - Read-only project mounts
    - Network restrictions
  </Step>

  <Step title="Rollback Validation">
    Verifies execution safety before allowing persistence:

    - Automatic cleanup
    - Comprehensive audit logging
    - Safety recommendations generated
  </Step>
</Steps>

### Safety Levels

Commands are classified into safety levels:

| Level | Patterns | Examples |
|-------|----------|----------|
| **Critical** | `rm -rf`, `curl.*\|.*bash`, `sudo` | File deletion, pipe to shell |
| **High** | `chmod`, `docker.*--privileged`, `eval` | Permission changes, privileged containers |
| **Medium** | `npm install -g`, `systemctl`, `wget` | Global installs, system services |
| **Low** | Standard commands | Regular development commands |

### Dagger Availability

The safety validator checks for Dagger availability:

```bash
# Check if Dagger is installed
dagger version
```

<Warning>
  If Dagger is not available, container validation is disabled. Install from [dagger.io](https://dagger.io) for full safety validation.
</Warning>

## Plugin Validation

Validates the plugin manifest and marketplace configuration:

```bash
bun run validate:plugin
```

**Checks:**

<Steps>
  <Step title="Plugin Manifest">
    Validates `.claude-plugin/plugin.json`:

    - Required fields: name, version, description, author
    - Valid semantic versioning
    - Agent references match actual files
    - Phase counts match actual command files
  </Step>

  <Step title="Marketplace Config">
    Validates `.claude-plugin/marketplace.json` (optional):

    - Plugins array exists
    - ccprompts plugin included
    - Valid configuration structure
  </Step>

  <Step title="Directory Structure">
    Validates symlinks and directories:

    - `commands/` symlink → `.claude/commands/`
    - `agents/` symlink → `.claude/agents/`
    - All 12 phase directories present
  </Step>

  <Step title="Documentation">
    Checks for required documentation:

    - README.md
    - PLUGIN.md
    - CLAUDE.md
  </Step>
</Steps>

## Troubleshooting Validation

### Command Count Mismatch

**Error:** `Expected 70 commands, found X`

**Fix:**

```bash
# Count actual command files
find .claude/commands -name "*.md" | wc -l

# Update .claude-plugin/plugin.json if adding/removing commands
# Update phase counts to match actual files

# Re-run validation
bun run validate
```

### Security Issues

**Error:** `SECURITY - Hardcoded API key detected`

**Fix:** Use placeholders for sensitive data:

<CodeGroup>
```javascript Bad
const API_KEY = "sk-1234567890abcdef";
```

```javascript Good
const API_KEY = "REPLACE_WITH_YOUR_API_KEY";
// or
const API_KEY = process.env.API_KEY;
```
</CodeGroup>

### Markdown Lint Errors

**Error:** Various markdown formatting issues

**Fix:**

```bash
# Auto-fix most issues
bun run lint:fix

# Check remaining issues
bun run lint
```
